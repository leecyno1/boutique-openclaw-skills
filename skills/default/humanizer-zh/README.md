# Humanizer-zh：去除AI写作痕迹

Humanizer-zh 是一个用于去除文本中 AI 生成痕迹的工具，帮助你将 AI 生成的内容改写得更自然、更像人类书写的文本。

## 安装

### 方法一：通过 npx 一键安装（推荐）

```
npx skills add https://github.com/idao-cube/humanizer-zh.git

# 选择你要使用的工具
# 默认安装 ~/.agents/skills
```

### 方法一：源码安装

```bash
# Claude
mkdir -p ~/.claude/skills
git clone https://github.com/idao-cube/humanizer-zh.git ~/.claude/skills/humanizer-zh

# Codex
mkdir -p ~/.codex/skills
git clone https://github.com/idao-cube/humanizer-zh.git ~/.codex/skills/humanizer-zh

# Project项目 （如：workspace/Project项目）
mkdir -p Project项目/skills
git clone https://github.com/idao-cube/humanizer-zh.git Project项目/skills/humanizer-zh
```

### 方法三：手动安装/更新（仅复制技能文件）

如果你已经克隆了这个仓库（或已下载 `SKILL.md`），把技能文件复制到 Claude Code/ Codex / Project项目 等 的 skills 目录：

```bash
# Claude
mkdir -p ~/.claude/skills/humanizer-zh
cp SKILL.md ~/.claude/skills/humanizer-zh/

# Codex
mkdir -p ~/.codex/skills/humanizer-zh
cp SKILL.md ~/.codex/skills/humanizer-zh/

# Project项目
mkdir -p Project项目/skills/humanizer-zh
cp SKILL.md Project项目/skills/humanizer-zh/
```

最终结构
```
~/.claude/skills/humanizer-zh/
├── SKILL.md       # 技能定义文件（中文版）
└── README.md      # 说明文档

~/.agents/skills/humanizer-zh/
├── SKILL.md       # 技能定义文件（中文版）
└── README.md      # 说明文档


```

## 使用方式

在 Claude Code / Codex / OpenCode / openclaw 中调用该技能：

1. 直接润色文本
   
```
/humanizer-zh
[在这里粘贴你的文本]
```

2. 在对话中使用

```
请用 humanizer-zh 帮我把这段文字改得更像人写的：[你的文本]
```

3. 处理文件内容

```
/humanizer-zh 请人性化 你的文章.md 文件中的内容
```

## 概述

该技能基于 [Wikipedia 的“Signs of AI writing”](https://en.wikipedia.org/wiki/Wikipedia:Signs_of_AI_writing) 指南，由 WikiProject AI Cleanup 维护。这份全面指南来自对数千条 AI 生成文本实例的观察。

技能还包含最后一轮“明显 AI 生成”审查，以及第二次改写，用于捕捉首稿中残留的 AI 味道。

### 来自 Wikipedia 的关键洞察

> "大型语言模型使用统计方法来预测下一句可能是什么。结果通常会趋向于在最广泛场景下都最可能成立的那种表达。"

## 检测到的 24 种模式（含前后对比例子）

### 内容模式

| # | 模式 | 修改前 | 修改后 |
|---|------|--------|--------|
| 1 | **意义夸大** | "标志着……演进中的关键里程碑" | "该机构成立于 1989 年，用于收集区域统计数据" |
| 2 | **靠媒体名头抬高重要性** | "被《纽约时报》、BBC、FT 和《印度教徒报》报道" | "她在 2024 年《纽约时报》采访中提出……" |
| 3 | **表层化的 -ing 分析** | "象征着……反映了……展示了……" | 删除空话，或补上真实来源与细节 |
| 4 | **宣传腔语言** | "坐落在风景如画、令人惊叹的地区" | "是贡德尔地区的一座城镇" |
| 5 | **模糊归因** | "专家认为它发挥着关键作用" | "根据 2019 年某机构调查……" |
| 6 | **模板化挑战句式** | "尽管面临挑战……仍持续繁荣" | 写出实际挑战及对应事实 |

### 语言模式

| # | 模式 | 修改前 | 修改后 |
|---|------|--------|--------|
| 7 | **AI 常见词汇** | "此外……见证了……格局……彰显了……" | "也……仍然常见" |
| 8 | **回避系动词** | "充当……具备……拥有……" | "是……有……" |
| 9 | **否定并列句式** | "这不只是 X，而是 Y" | 直接陈述观点 |
| 10 | **三项并列执念** | "创新、启发与洞见" | 使用自然数量的列举 |
| 11 | **同义词轮换** | "主角……核心人物……关键角色……英雄" | "主角"（最清晰时可重复） |
| 12 | **伪范围表达** | "从宇宙大爆炸到暗物质" | 直接列出具体主题 |

### 风格模式

| # | 模式 | 修改前 | 修改后 |
|---|------|--------|--------|
| 13 | **破折号滥用** | "是机构\-\-不是个人\-\-但这种情况仍在继续\-\-" | 改用逗号或句号 |
| 14 | **粗体滥用** | "**OKRs**、**KPIs**、**BMC**" | "OKRs、KPIs、BMC" |
| 15 | **行内标题式列表** | "**绩效：** 绩效有所提升" | 改写为自然行文 |
| 16 | **标题式大小写标题** | "战略谈判与合作伙伴关系"（每个词都大写） | "战略谈判与合作伙伴关系"（常规句式） |
| 17 | **表情符号** | "🚀 发布阶段：💡 关键洞察：" | 删除表情符号 |
| 18 | **弯引号** | `他说：“这个项目”` | `他说: "这个项目"` |

### 沟通模式

| # | 模式 | 修改前 | 修改后 |
|---|------|--------|--------|
| 19 | **聊天机器人痕迹** | "希望这对你有帮助！如果你愿意我还可以……" | 完全删除 |
| 20 | **信息截断式免责声明** | "由于可用来源有限，细节暂不明确……" | 找到来源或删除该句 |
| 21 | **讨好式语气** | "这个问题太棒了！你完全正确！" | 直接回应问题 |

### 填充词与过度保留

| # | 模式 | 修改前 | 修改后 |
|---|------|--------|--------|
| 22 | **填充短语** | "为了能够", "由于这样的事实" | "为", "因为" |
| 23 | **过度保留措辞** | "可能也许大概会" | "可能" |
| 24 | **空泛结论** | "未来一片光明" | 给出具体计划或事实 |

## 完整示例

**修改前（AI 味较重）：**
> 这个问题很棒！下面是一篇关于这个主题的文章，希望对你有帮助！
>
> AI 辅助编程是大型语言模型变革潜力的有力见证，标志着软件开发演进中的关键时刻。在当今快速变化的技术格局中，这些处于研究与实践交汇点的突破性工具，正在重塑工程师构思、迭代和交付的方式，凸显其在现代工作流中的核心作用。
>
> 从本质上看，它的价值主张很清晰：简化流程、增强协作、促进一致性。这不只是自动补全，更是在规模化释放创造力，确保组织在保持敏捷的同时，为用户交付流畅、直观且强大的体验。这个工具充当催化剂。助手扮演合作伙伴。系统成为创新的基础。
>
> 行业观察者指出，这项技术的采用已从爱好者实验加速走向企业级落地，从独立开发者扩展到跨职能团队。该技术曾被《纽约时报》、Wired 和 The Verge 报道。此外，其在生成文档、测试和重构方面的能力，也展示了 AI 如何促进更优结果，凸显自动化与人类判断之间的复杂互动。
>
> - 💡 **速度：** 代码生成显著更快，减少摩擦并提升开发者效率。
> - 🚀 **质量：** 通过改进训练，输出质量得到提升，有助于提高标准。
> - ✅ **采用：** 使用量持续增长，反映出更广泛的行业趋势。
>
> 虽然基于现有信息可获得的细节有限，但仍可认为这些工具可能带来一定积极影响。尽管存在新兴技术常见挑战，包括幻觉、偏见与责任归属问题，这一生态仍在持续发展。为了充分释放其潜力，团队需要与最佳实践保持一致。
>
> 总之，未来前景光明。随着我们继续迈向卓越，令人兴奋的阶段正在到来。如果你希望，我可以进一步展开其中任一部分。

**修改后（更像人写的）：**
> AI 编程助手能加快工作里最枯燥的部分。它很擅长样板代码，比如配置文件和你不想手写的那些胶水代码。它也能帮你先搭出测试草稿，但你还是得自己逐行看。
>
> 真正危险的是，它给出的建议看起来总是很笃定。我也接收过那种能编译、能过 lint 的代码，后来才发现完全跑偏了，因为我当时没有认真审。
>
> 如果你把它当自动补全，并且每一行都审，它就很有用。要是你用它来省掉思考，它只会帮你更快把 bug 发到线上。
>
> 真正的兜底只有测试。没有测试，你基本只能靠感觉判断。
